TL;DR

Our harness is in the right architectural neighbourhood for what it says on the tin: a learning-grade reproduction of Codex patterns in Effect-TS. The shape we ended up with — recursive-stream turn loop, swappable approval as an Effect Context.Tag, tagged error unions, separable Tool / Toolkit / LanguageModel layers — is closer to opencode than to anything else in repos/.

Where the comparison gets uncomfortable is that we don't have a system prompt at all, our shell tool ignores cancellation, and our approval gate is coarser than even pi's per-call callback. Those three things are short patches.

The whole review came from reading our own packages/ source and the vendored repos/ subtrees side by side, axis by axis. Everything below has a file:line citation behind it; I'm just not going to spam them all back into prose.

The four projects, briefly

  • effectclanker Us. TypeScript + @effect/ai. Six tools wired to Anthropic, hand-rolled recursive-stream loop, Ink TUI. Four packages, strict downward layering.
  • codex OpenAI's coding agent. Rust core, Ratatui TUI. WebSocket+SSE streaming, OS sandboxing, sub-agent "guardian" for approval review, full MCP consumer.
  • pi Earendil's coding agent. TypeScript. Event-stream protocol, JSONL session persistence with branching, hook-everything extensibility model.
  • opencode A mature open-source agent. TypeScript, Effect-based. Plugin hooks, pattern-based approval rulesets, multi-provider routing, MCP, compaction.

Tool specs: how the model sees a capability

All four projects model tools as a name + parameter schema + success/failure types plus an executor. The interesting question is what else sits on the spec — the metadata that the runtime uses but the model doesn't see directly.

project schema per-tool metadata MCP-style annotations
us effect/Schema failureMode: "return" only no
codex JSON schema via codex_tools::ToolSpec supports_parallel_tool_calls, exposure, SearchInfo, pre/post hook payloads yes
pi typebox executionMode: "sequential" | "parallel", prepareArguments shim no
opencode Effect Schema, plus a dynamic JSON-Schema mode none structural; permissions live at session level yes (dynamic mode)

Our slimness here is mostly fine — we own six tools and we don't fan calls out in parallel, so codex's supports_parallel_tool_calls flag would be premature. But there's one thing we got right that the others didn't: typed failure schemas.

Codex's tool failure path is FunctionCallError::RespondToModel(String) — a stringified message goes back to the model. Opencode's ToolFailure is the same shape. Pi encodes failures into the message protocol as text. We pipe a Schema.TaggedError union back as structured JSON. The model sees { "_tag": "FileNotFound", "path": "..." } instead of "file not found: ...". Better for the model, better for tests. Keep this.

The turn loop

Every one of these projects rolls its own turn loop. We did too: packages/harness/src/agent-loop.ts is a recursive Stream that streams the model, dispatches tool calls, and recurses. The shape is a direct port of opencode's tool-runtime.ts.

concern us codex pi opencode
iteration bound stepCountIs(25) token budget + final-message shouldStopAfterTurn callback stopWhen callback
cancellation none observable CancellationToken end-to-end AbortSignal per tool Effect interruption via Streams
compaction none compact.rs on threshold logged entry, applied on resume compaction.ts on overflow

Two gaps surface here. First, no cancellation: if you Ctrl-C our TUI while the shell tool is running npm install, the process exits but the child keeps going to completion. That's a leak. Effect already does interruption; we just don't thread it into tool handlers.

Second, no compaction: we'll hit Anthropic's context window on long sessions and silently fail. Opencode's compaction is the cheapest reference to copy — markdown summary of older turns plus a preserved tail of N most recent.

Approval & safety

We have four approval variants — Auto, Deny, Interactive, Ink — gated at four tool-class discriminants (shell | write | edit | apply_patch). Everything else is implicit-allow (read, grep, glob).

That's a coarse gate. Auto-approve git status but ask for rm is not something you can express. The cleanest upgrade comes from opencode's evaluate.ts: a ruleset of { permission, pattern, action } entries with wildcard match, evaluated last-rule-wins. A direct port; ~80 lines.

What we got right here that nobody else did: approval is a Context.Tag Layer. Swapping auto-approve for interactive for Ink is one line at the runtime boundary. Codex bakes approval into the session; opencode bakes it into rulesets; neither has this clean a seam. Don't refactor that away.

Codex goes further still: OS-level sandboxing via seatbelt on macOS, landlock on Linux, AppContainers on Windows. Worth understanding, almost certainly not worth porting for a learning-grade harness.

Conversation state

Our state lives in Chat.Service (in-memory) plus a ChatStateController for the TUI. Nothing is persisted. Exit the CLI; the conversation is gone.

Pi is the most interesting comparison here. Pi's conversation is a tree, stored as JSONL on disk, with a typed entry union (MessageEntry | CompactionEntry | BranchSummaryEntry | …). Resumption re-parses the JSONL. Branching creates a new session with a parent pointer. It's the kind of design that pays off the moment you want to compare two attempts at the same task.

For a learning harness this gap is defensible — the CLI's one-shot mode masks the missing persistence. But it's the single biggest UX cliff for anyone using chat for more than a few prompts.

Error handling

Our strongest axis. Tagged Effect unions in packages/tools/src/errors.ts, mapFileError discriminates PlatformError by enum, and every handler surfaces structured failure via failureMode: "return". The model never sees a stack trace.

What we don't have is a retry policy. Opencode's retry.ts is the right shape to copy: classify by status code & rate-limit pattern, exponential backoff capped at 30s, respect retry-after headers. Cheap to copy, big payoff the first time you see a 429.

The missing system prompt

This is the punchline. We instantiate Chat.empty in both cli.ts and chat-runtime.tsx with no system prompt at all. The model is operating without knowing:

  • what OS it's running on
  • what directory it's in
  • whether there's a git repo (and what state it's in)
  • what date it is
  • what its tools do beyond the one-line descriptions

All three reference repos inject this. Opencode's version is a one-screen Effect that yields a context block: 

environment(model)
  → "model: claude-haiku-4-5
      working directory: /Users/.../effectclanker
      worktree root: /Users/.../effectclanker (clean)
      platform: darwin 25.2.0
      date: 2026-05-14"

Codex layers on top of that — base instructions, developer instructions, skill injections, plugin injections. We don't need any of that yet. We just need a system prompt that exists.

Of every gap surfaced by this review, this is the highest-ROI single change. Probably an evening of work.

What we got right (don't regress)

  • Effect-first error handling. Structured Schema.TaggedError unions surfaced through failureMode: "return" beat all three references' string-error patterns.
  • Approval as a swappable Layer. No reference has this clean a seam.
  • Hand-rolled recursive-stream loop. Matches opencode's shape and keeps us in control of cancellation, compaction, and hooks when we get there.
  • Per-package boundary tests. The test/package-boundary.test.ts in every workspace enforces dependency direction. None of the three references enforce layering this strictly.
  • Documenting apply_patch as a Codex-grammar mirror rather than inventing a new diff format. The lineage is explicit; traceability beats "improvement" here.

What I'd do next, ranked

Top ROI System prompt

Static context (cwd, OS, date) injected into Chat.empty. The model immediately gets dramatically more useful. Reference shape: repos/opencode/packages/opencode/src/session/system.ts:48-63.

Top ROI AbortSignal through handlers

Make runAgentTurn interruptible and propagate to the shell handler so Ctrl-C actually kills npm install. Pi's execute(toolCallId, params, signal?, onUpdate?) signature is the model.

Top ROI Pattern-based approval rules

Replace the four-discriminant enum with { tool, pattern, action } evaluated by wildcard. Direct port of opencode's evaluate.ts:9. Per-command granularity unlocks practical auto-approve.

Medium Retry policy

Exponential backoff with header awareness for 429s and 5xx. Port from opencode's retry.ts.

Medium Compaction

Markdown summary plus last-N tail when the context window is close to full. Required for the chat command to be useful past ~50 turns.

Medium Hook seam in the toolkit layer

Before/after callbacks would unlock logging, redaction, and per-tool approval rewriting without touching individual handlers.

Skip for now OS sandbox, multi-provider, MCP, on-disk sessions

All four are valuable, all four are project-sized, and the learning-grade scope is best served by the smaller items first. They're already in docs/backlog.md with codex cross-references when we get there.
