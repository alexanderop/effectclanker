How a Coding Agent Works — and How to Build One with Effect

Coding agents feel like sorcery. They're not. They're a loop, six functions, and one nasty default flag that will eat your afternoon if you let it.

The whole trick, in one paragraph

Have you ever had the unsettling experience of watching a coding agent edit a file, run your tests, read the error, and then fix its own bug? It feels like the model is doing something intelligent. It is — but the intelligence is narrower than it looks.

A coding agent is a language model that's been given a list of functions it can ask you to run, plus a loop that keeps calling it until it stops asking. The model never touches your filesystem. It can't. It just emits JSON that says "please run read('/etc/hosts')" — and your program decides whether to honor that, runs the function, and hands the result back as the next message.

Strip away the framework and a coding agent is six tools, one loop, and a lot of trust. Everything else is engineering taste.

The shape of one turn

Let's look at the round-trip first, before we worry about loops. One prompt goes in. Somewhere along the way a real file gets read. A response comes out. The interesting thing — the thing the model can't see — is everything in the middle.

The model never reads the file. It asks. Your program reads the file and hands the bytes back. That asymmetry — model in the sandbox, your handler in the dirty world — is the whole security model and the whole programming model at once.

Three primitives are enough

OK so here's the part that took me an embarrassingly long time to internalize. The Effect-AI authors decomposed "the agent" into three tiny pieces, and the rest of the library is — honestly — just plumbing around them.

That split is load-bearing. Specs live next to the model's view of the world. Handlers live next to the filesystem. They never touch each other directly — the Toolkit is the only place the two meet. Once you see it, you'll see it everywhere.

A Tool is just metadata

A Tool is just metadata. Really. I know that sounds like a deflection, but stick with me — this was the thing that took me longest to internalize, and it's the entire reason @effect/ai is so pleasant to use.

You write a spec in TypeScript. The framework converts it to JSON Schema. The model sees the JSON. Edit the inputs below and watch your spec on the left turn into the schema on the right — that's literally all that's happening at this layer.

The first time I saw this, I assumed I was missing something. Surely somewhere I have to wire the spec to the actual code? You do — but that wiring lives one layer up. The spec itself knows nothing about the filesystem, knows nothing about fs.readFile, and frankly knows nothing about anything. It's a description.

A Toolkit is where specs meet code

OK, so we've got specs. Specs don't do anything. Time to give them implementations. Toggle a tool on or off and watch both halves of the wiring update in lockstep — the registry on the left, the handler record on the right:

The handler record's type is derived from the Toolkit.make call. Rename a tool, forget a handler, pass a handler whose return shape disagrees with its tool's success schema — TypeScript stops compiling. The check is baked into the type system, and you get it for free.

This is the Effect part of "building it with Effect" that's hard to show in a screenshot. The Toolkit isn't a clever runtime — it's a type that knows the names and shapes of every tool, and forces your handler record to match. The runtime is barely there.

The loop you have to write yourself

This one bit me, and it bites everyone, so let's just confront it. generateText is not the agent loop. It's one model call plus intra-call tool dispatch. If you want the model to see the tool results and respond to them, you have to call generateText again with an updated prompt.

Or use Chat from @effect/ai, which keeps the history in a Ref for you and re-invokes generateText per call. But even Chat doesn't loop on its own — it's still you, deciding when to stop. An agent is the place where the human's judgment about when we're done hasn't been automated yet.

The one default that will eat your afternoon

OK. Real talk. This one bit me, and I want to spare you the same hour I lost. Tool.make has a field called failureMode. It defaults to "error". And that default — for almost every tool you'll ever write — is wrong.

Notice which one feels like an agent and which one feels like a 500. A coding agent that crashes the moment the model tries to read a file that doesn't exist isn't really an agent — it's a very expensive autocomplete. Set failureMode: "return" whenever you declare a non-Never failure schema. The two go together, always.

That's it. You can build one now.

Three primitives — Tool, Toolkit, LanguageModel — and a loop you write yourself with while or Effect.iterate or whatever else you fancy. The model never touches your filesystem; your handlers do. The framework converts your specs to JSON Schema, dispatches calls by name, encodes results back as data.

Every coding agent you've ever used — Cursor, Codex, Claude Code, the one in your editor right now — is some version of this. They differ in the size of the tool list, the policy around approval, the cleverness of the prompt. But the bones are the same. And once you've seen the bones, you can build your own in an afternoon. Really.
